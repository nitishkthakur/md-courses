01 · First principles

Why memory exists at all

Start from the metal. A large language model is a pure, stateless function. Each call computes output = f(context) and then forgets everything. There is no hidden variable carried between calls, no register that survives the forward pass. The weights are frozen at inference; the only thing the model "knows" in the moment is the sequence of tokens you hand it — the context window.

Everything we call "memory" is therefore an illusion engineered around a stateless function: an external apparatus that makes f behave as if it had persistent state. That apparatus has exactly one channel into the model — it must take stored information, select the right pieces, and write them back into the context window before the next call. This single constraint generates the entire field.

Two foundational terms

Context window — the bounded token buffer the model attends to on a given call. It is the agent's working memory: fast, fully accessible, but volatile (gone after the call) and scarce (fixed size, and, as §02 shows, degrading well before it fills).

External / long-term memory — anything stored outside the window (files, databases, vector indexes, graphs) that must be explicitly retrieved and re-injected to have any effect. The model cannot "recall" it the way a human does; a tool call or a retrieval step has to physically place it in context.

The 2025–26 surveys converge on a clean way to describe the machinery: agent memory is a write → manage → read loop coupled to the agent's perception and action. You decide what to write out of the conversation, how to manage it once stored (organise, merge, forget), and how to read the right slice back when it is needed. Keep that loop in mind; §04 dissects each verb.

02 · The motivating constraint

Why not just use a giant context window?

This is the question every interviewer asks, and the answer is the intellectual core of the field. If 1M- and 2M-token windows exist, why bother with memory engineering at all? Because a long window is necessary but not sufficient, for four independent reasons.

1 · Context rot — longer context is measurably worse

Chroma's 2025 context rot study tested 18 frontier models (including GPT-4.1, Claude 4, Gemini 2.5, Qwen3) and found that every one degrades as input length grows — not at the window limit, but continuously. Strikingly, even when distractors are stripped out and only total length is varied, accuracy still falls (~8% on average from length alone). Long context is intrinsically harder regardless of what fills it. Practitioners now budget an effective context window that is often 4–10× smaller than the advertised maximum.

2 · Lost-in-the-middle — position is not neutral

The Liu et al. "lost in the middle" result (Stanford/TACL) shows models attend well to the start and end of context and poorly to the middle — a 30%+ accuracy drop for facts buried mid-prompt. Bigger windows pushed the problem further out but did not remove it. So even with perfect retrieval, where you place evidence matters.

3 · The attention budget is quadratic

Transformer self-attention forms n² pairwise interactions for n tokens: 100K tokens ⇒ ~10 billion relationships. Anthropic frames this as a finite attention budget — every token you add depletes the model's ability to focus on the rest. Models also see far more short sequences than long ones in training, so they have fewer specialised parameters for long-range dependence. The result is a performance gradient, not a hard cliff.

4 · Cost, latency, and impermanence

Tokens cost money and time linearly; a fat prompt on every turn is expensive and slow. And the window is ephemeral — it vanishes when the session ends. None of "what the user told me last week", "what I tried yesterday that failed", or "the conventions of this codebase" survive in a window. Persistence across sessions is something only external memory provides.

Interview answer · "Why not just use long context?"

Compress it to: "Long context is real but it rots — accuracy degrades with length and with mid-prompt position (Chroma context-rot, lost-in-the-middle), attention is quadratic so there's a finite attention budget, and the window is both costly per-token and ephemeral across sessions. So you treat context as a scarce, decaying resource: keep the smallest set of high-signal tokens hot, push the long tail to external memory, and retrieve just-in-time." That last sentence is the thesis of the whole discipline.

This is why the field converged on a single architecture by late 2025: working memory inside the window, the long tail outside it; and for stable, frequently-used knowledge, compiled knowledge (a curated wiki / set of cards) often beats live retrieval — Karpathy's "LLM wiki" point. The discipline that does this curation has a name: context engineering.

Term · context engineering

Anthropic's framing (Sept 2025): the successor to prompt engineering. Where prompt engineering writes a good instruction once, context engineering is the iterative practice of curating the optimal set of tokens at every step of inference — system prompt, tools, examples, message history, retrieved data, and memory. The guiding principle: find the smallest set of high-signal tokens that maximises the probability of the desired outcome. Memory is the part of context engineering concerned with persistence.

One orthogonal axis matters too. The surveys split all memory into two kinds by where it physically lives:

  • Parametric memory — knowledge baked into the model's weights (what fine-tuning or pre-training stores). Fast at inference, but opaque, hard to edit, hard to attribute, and frozen until you retrain.

Most deployed agents lean heavily non-parametric, precisely because you can audit it, correct it, and delete it on request. 04 · The operations

The verbs: write, manage, read, forget

Every memory system — simple or exotic — is some implementation of four operations. Naming them lets you reason about where a system is sophisticated and where it is naïve.

① Write (encode)

Two decisions: what to store and when. On what: store raw turns (cheap, noisy) or LLM-extracted facts (clean, costs an extraction call). On when, the LangMem distinction is the one to know:

  • Hot-path writing — the agent updates memory in the loop, before responding. Immediate, but adds latency and tokens to every turn.
  • Background writing — a separate process extracts and consolidates memories asynchronously after the turn (or on a schedule). No user-facing latency; this is the basis of "sleep-time" memory (§08).

② Manage (consolidate, index, update)

Raw writes accumulate into noise and contradiction — the memory hoarding problem. Management fixes this:

  • Consolidation / dedup / conflict resolution. Mem0's update step is the clearest pattern: an LLM decides, for each new fact, whether to ADD (new), UPDATE (augment), DELETE (contradicted), or NOOP (already known).
  • Reflection / summarisation. Periodically compress many low-level memories into higher-level insights (the generative-agents move, §07).
  • Indexing. Build the structures retrieval will use — embeddings, inverted indexes, graph edges. Often the operation surveys say is most under-built.

③ Read (retrieve)

Selecting the right slice to re-inject. The methods, cheapest to richest:

  • Dense — embed query & memories, return top-k by cosine similarity (vector search). Captures meaning, misses exact terms.
  • Sparse — keyword/lexical match (BM25). Captures exact terms (IDs, error codes, column names), misses paraphrase.
  • Hybrid + rerank — run both, fuse, then re-rank with Reciprocal Rank Fusion (RRF), Maximal Marginal Relevance (MMR, for diversity), or a cross-encoder. This is the production default.
  • Graph traversal — follow edges from seed nodes (your filter-then-traverse).
  • Scored retrieval — combine signals like recency, importance, relevance into one rank (generative agents, §07).
  • Iterative / agentic retrieval — the agent retrieves, reasons, then retrieves again, walking toward the answer rather than one-shotting it.

④ Forget (decay, evict, invalidate)

Forgetting is a feature, not a bug — it keeps memory relevant and bounded. Three flavours: time decay (down-weight or expire old items, à la MemoryBank's Ebbinghaus-style forgetting curve), eviction (drop low-value items under a budget), and — the most elegant — invalidation: when a fact changes, don't delete the old one, mark it no longer valid as of date X. That preserves a queryable history (Zep/Graphiti, §06) and is exactly what a regulated audit trail wants.

Pitfall · the naïve system has no "manage" and no "forget"

The most common failure is "append-only RAG over chat history": write everything, never consolidate, never forget. It bloats, contradicts itself, and retrieves stale facts with confidence. When you assess any memory product (or your own), ask first: how does it consolidate, and how does it forget? If the answer is "it doesn't," it is a log, not a memory.

05 · Short-term toolkit

Short-term / working-memory techniques

These manage the hot tier — the window — over a single long-running task. For a deep-work coding agent like clicode this tier is where most of the leverage is, because the failure mode isn't "the model is dumb," it's "the context got polluted." Roughly in order of sophistication:

naïve
Full history — keep everything; blows the window
cheap
Sliding window / truncation — keep last N turns
better
Summary buffer — replace old turns with a running summary
strong
Compaction — re-init window from a high-fidelity summary

Compaction

The first serious lever, and what Claude Code uses. As the window nears its limit, summarise its contents and start a fresh window from that summary. The craft is in the selection function: Claude Code's compaction preserves architectural decisions, unresolved bugs, and implementation details, while discarding redundant tool outputs, then carries forward the few most-recently-touched files. Anthropic's tuning advice is worth stealing verbatim: maximise recall first (capture everything that might matter), then iterate for precision (cut the superfluous). The lightest-touch form is tool-result clearing — once a tool result is deep in history, the raw blob rarely needs to stay.

Structured note-taking ("agentic memory")

The agent writes notes to a file outside the window and reads them back later — a NOTES.md, a to-do list, a progress ledger. This is the cheapest possible long-term memory and it is shockingly effective: in Anthropic's "Claude plays Pokémon" demo the agent maintains tallies and maps across thousands of steps and resumes coherently after each context reset purely from its own notes. For clicode this is the single highest-ROI addition you don't yet have (see §09).

Just-in-time retrieval & progressive disclosure

Instead of pre-loading everything, keep lightweight identifiers in context — file paths, queries, links — and load the actual content on demand via tools. Claude Code drops CLAUDE.md in up front but uses glob/grep to pull files just-in-time, sidestepping stale indexes. The metadata of the identifiers is itself signal: a file's name, folder, and timestamp tell the agent whether to open it. Progressive disclosure is the incremental version — reveal structure layer by layer, loading only the next needed piece.

Career bridge · you invented this twice

Your EUCT Excel MCP server is a progressive-disclosure engine: it returns truncated "patches" with Patch IDs, and the agent decides which to fetch in full on a second pass — fitting a 60-sheet workbook into context the way Anthropic describes fitting a large database into Claude Code. And BMG's MCP-over-graph with recursive column tracing is JIT retrieval with lightweight identifiers. In an interview, name them as such — you'll be describing 2025-frontier patterns you shipped in production before they had blog posts.

Sub-agent architectures

Rather than one agent holding the whole task, spawn specialised sub-agents with clean, isolated context windows. Each can burn tens of thousands of tokens exploring, but returns only a distilled 1–2K-token summary to the lead agent. The detailed search context stays quarantined; the lead stays focused on synthesis. Anthropic reports this beats single-agent setups on complex research. The trade-off: coordination overhead and the risk that the summary drops something the lead needed.

Interview framing · the three long-horizon levers

Anthropic names three techniques for tasks exceeding the window — compaction, structured note-taking, sub-agents — and which to use depends on the task: compaction for conversational back-and-forth, note-taking for iterative work with clear milestones, sub-agents for parallel exploration. Being able to say "I'd reach for note-taking + sub-agents here because the work is milestone-based with verbose exploration" signals real fluency.

06 · Long-term representations

Long-term memory: how to represent it

The hot-tier techniques buy you a long single session. To persist across sessions and scale, you need a durable representation. This is the part most connected to your graph and GNN background — and the choice of representation is the single biggest architectural decision in a memory system.

(a) Flat vector store + RAG — the workhorse

Chunk your corpus, embed each chunk, store vectors in an index, retrieve top-k by similarity, stuff into context. Simple, scalable, and the right default for large static document sets. But note the crucial distinction the field insists on:

Term · RAG is a tool for memory, not memory itself

Classic RAG retrieves over a static corpus. Memory is about an evolving store that the agent writes to, consolidates, and invalidates over time. RAG is the read mechanism you might plug into a memory system; it is not, by itself, memory. Conflating the two is a common interview tell.

The infrastructure layer here is the vector-database market: pgvector (Postgres extension — start here if you already have Postgres), Chroma (lightweight, local-first), Qdrant, Weaviate, Milvus (scale), Pinecone (managed), and FAISS (a library, not a server). They are largely interchangeable plumbing; the interesting decisions are upstream (what to embed, how to chunk) and downstream (how to rerank).

(b) Hierarchical summary trees

Build a tree by recursively clustering and summarising chunks (RAPTOR), so retrieval can pull either a fine detail (leaf) or a high-level gist (interior node). MemGPT's recursive summary of overflowed history is the same idea in miniature. Good when answers need varying levels of abstraction.

(c) Knowledge graph — entities & relations

Represent memory as nodes (entities) and labelled edges (relations): an LLM extracts (subject, relation, object) triplets from text and inserts them. Mem0g does exactly this; Microsoft's GraphRAG additionally clusters the graph into thematic "communities" and pre-summarises them for global "what are the themes?" queries. Strength: multi-hop and relational reasoning ("how does column X depend on source Y three transforms back?"). Cost: extraction is lossy and LLM-dependent. This is your home turf — it is the same object as the BMG DAG.

(d) Temporal / bi-temporal knowledge graph — the state of the art

Zep, built on the open-source Graphiti engine, is the system to know here. Its signature is the bi-temporal model: every edge records both when the fact was true in the world (t_valid / t_invalid) and when the system learned it (t_created / t_expired). When new information contradicts an old fact, Graphiti invalidates the old edge (closes its validity window) rather than deleting it. Three consequences fall out for free:

  • Contradiction-free answers — the agent reasons over current state, never forced to choose between a stale and a fresh fact.
  • Point-in-time queries — "what was the plan in January?" is answerable, not just "what is it now?"
  • Auditable provenance — every fact traces back to the source episode that produced it.

Zep organises memory in three tiers — episodic raw messages, extracted semantic entities/facts with temporal validity, and community summaries — and retrieves with a hybrid of cosine + BM25 + graph traversal plus reranking. It reports beating MemGPT on the DMR benchmark (94.8% vs 93.4%) and gains up to 18.5% on the harder LongMemEval. For a finance audit context, "memory with bi-temporal provenance you can replay to any date" is an extremely strong story.

(e) Self-organising note network — A-MEM (Zettelkasten)

A-MEM applies the Zettelkasten method: each memory is an atomic note with structured attributes (content, timestamp, LLM-generated keywords, tags, a contextual description, an embedding, and links). When a new note is added, the system autonomously generates links to related historical notes and may evolve those older notes — updating their context as new knowledge arrives. The structure isn't fixed in advance; it grows organically, agent-driven. Conceptually it's a knowledge graph that builds and rewires itself.

07 · The landscape

The systems landscape — named tools

Now the concrete frameworks, each with its philosophy. The most important conceptual split is memory layer (a bolt-on you add to any agent) vs agent runtime (your agent lives inside it) vs primitives (you wire the pieces yourself).

MemGPT / Letta — "the LLM as an operating system"

The foundational systems paper (Packer et al., 2023). The analogy: treat the context window as RAM and external stores as disk, and give the model OS-style tools to page information between them. Letta (the framework MemGPT became) makes memory tiers explicit: core memory (small, always-in-window, self-editable blocks — e.g. a persona and key user facts), recall memory (searchable conversation history), and archival memory (arbitrary long-term store). The defining move is self-editing memory: the agent calls tools like memory_insert/memory_replace to rewrite its own memory in the normal loop, and when the window fills it auto-compacts into a recursive summary. Letta's philosophy is persistence by default — you have to actively avoid remembering. Its newer "Letta Code" pushes this into coding agents with git-backed memory, skills, and subagents — a direct cousin of clicode.

Mem0 — "the memory layer"

Framework-agnostic: you bolt it onto LangChain, CrewAI, a custom loop, anything. Two phases — extraction (an LLM pulls salient facts from the conversation) and update (the ADD/UPDATE/DELETE/NOOP decision against existing memory). A graph variant, Mem0g, stores entity/relation triplets for relational queries. Mem0 reports strong LOCOMO numbers — ~26% relative gain over OpenAI's memory on an LLM-as-judge metric, with large latency and token savings versus full-context — because it sends compact extracted memories instead of whole histories.

Zep / Graphiti — "temporal memory at enterprise scale"

Covered in §06(d). Position it as: the choice when facts change over time and you need provenance and governance — bi-temporal graph, sub-200ms retrieval, SOC2/HIPAA posture. Graphiti is the open-source engine you can self-host on Neo4j/FalkorDB; Zep is the managed platform on top.

LangMem / LangGraph — "the primitives"

Not a runtime — a toolkit. LangGraph cleanly separates two storage roles you must never conflate: the checkpointer (in-thread conversation/session state) and the BaseStore (cross-thread, user-scoped long-term facts). On top, the LangMem SDK supplies the three memory types (semantic/episodic/procedural) and the two write mechanisms (hot-path tools vs background managers), with automatic consolidation and namespacing (every memory is scoped, typically by user_id, to prevent cross-user leakage). Note: its background-manager path is batch-oriented (high p95 latency) — use it asynchronously, not on the hot path.

The Anthropic / Claude Code stack — the one clicode mirrors

Because clicode is "Claude Code lite," this stack is your reference design. It has distinct, well-separated layers:

  • CLAUDE.md — the explicit, human-written instruction layer (procedural + semantic): build commands, conventions, architecture, "always do X." Loaded at session start, hierarchically (root files in full, subdirectory files on demand), supports @imports. Treated as context, not enforced policy — phrasing matters, and emphasis markers (IMPORTANT, YOU MUST) measurably improve adherence. Keep it tight (teams target <200, even ~60 lines) because every line is reloaded every turn.
  • Auto memory — the learned-pattern layer. Claude writes back durable, non-obvious things it discovers (an undocumented command, an environment-specific error fix) to a machine-local MEMORY.md, so the next session starts already knowing them. This is the agent becoming a contributor to its memory, not just a consumer.
  • Project rules (.claude/rules/*.md with paths: globs) — path-scoped instructions that load only when relevant files are touched. Pure just-in-time procedural memory.
  • Skills (SKILL.md) — packaged repeatable methodology, invoked by name or auto-loaded when relevant. Your BMG skill files, exactly.
  • Subagents — isolated context windows that return summaries; can carry their own scoped auto memory.
  • Compaction — automatic when the window fills (older tool outputs cleared first, then the conversation summarised).
  • The Memory Tool (API) — for building this into your own agents: a file-based cross-session memory (type: "memory_20250818") exposing create/read/str_replace/delete-style operations over a memory directory. This is the literal primitive you'd wire into clicode.

08 · The frontier

Where memory is headed (2026 and beyond)

You asked specifically where this is going. The research front in late-2025/2026 clusters into six directions — each is a strong "what's next" talking point.

1 · Sleep-time / background consolidation

Move memory formation off the critical path: while the agent is idle, a background process reviews recent activity, consolidates, reflects, and reorganises — like sleep consolidating a day's experiences. Letta formalises this as sleep-time compute with dedicated sleep-time agents; Anthropic has previewed background self-improvement in a similar spirit (early reports of agents reviewing past sessions to extract patterns). The win is non-blocking memory and steadily improving recall without per-turn latency.

2 · Learning what to remember (memory + RL)

Today's write/forget policies are mostly hand-tuned heuristics. The frontier learns them. A wave of work — Mem-α, MemAgent, MEM1 — treats memory construction as a reinforcement-learning policy: reward the agent for remembering what later proves useful and discarding what doesn't. Memory stops being a fixed pipeline and becomes a trained skill. (This is a natural bridge for someone with an optimisation/estimation background — it's a policy-learning problem over a memory state.)

3 · Episodic memory as "the missing piece"

A prominent position paper argues that episodic memory — concrete, replayable past experiences — is the under-served tier and the key to genuine long-horizon competence. Most systems over-invest in semantic facts and under-invest in "remember the episode where this exact situation arose and what happened." Expect richer episodic stores and experience-replay-style retrieval. (Directly relevant to clicode's gap — §09.)

4 · Self-evolving, experience-driven agents

Procedural memory that rewrites itself: agents that distil successful trajectories into reusable skills and improve from their own "lived experience" (the Voyager skill-library lineage; "Skill-RL"-style work; Letta's explicit thesis of agents that learn from work). The endpoint is an agent whose competence compounds with use rather than resetting each session.

5 · Multi-agent shared memory

As systems become teams of agents, memory becomes shared infrastructure with the attendant hard problems: consistency, who-can-read-what access control, and knowledge transfer between agents (G-Memory, LEGOMem, collaborative-memory work). Memory turns into a governed, multi-tenant data system — squarely an enterprise/finance concern.

6 · Governance, safety, and evaluation maturing

Three threads worth tracking, all enterprise-relevant: memory-injection attacks (adversaries planting false memories — a real attack surface), provenance & right-to-be-forgotten (auditable, expungeable memory — the bi-temporal/non-parametric argument again), and better evaluation. On evaluation specifically:

Term · the memory benchmarks (and their critique)

LoCoMo — very-long-term conversational memory (up to ~35 sessions, 300+ turns). LongMemEval — long-term interactive memory with hard temporal reasoning, closer to enterprise use. DMR (Deep Memory Retrieval) — the older MemGPT-era metric. MemBench — broader coverage. The sharp critique from the 2026 surveys: classic IR metrics (recall@k, nDCG) tell you whether the right item was retrieved, but say nothing about whether the agent used it correctly, or about staleness, contradiction, forgetting quality, and governance. Memory evaluation must judge decision quality, not just retrieval quality. Quoting this distinction is a strong signal of depth.

The meta-trend

Two forces pull in opposite directions. Bigger, cheaper context windows reduce some need for retrieval. But context rot, cost, governance, and cross-session persistence mean the need never vanishes — it moves. As models get smarter they need less prescriptive memory engineering and can manage more of their own context. Anthropic's own bet: "do the simplest thing that works" — start minimal, add memory machinery only where a real failure mode demands it.

Pitfall · don't over-retrieve into clicode

The instinct on a 7.6M-token estate is to pull more context. Context rot says the opposite: pull the smallest high-signal set. Your batched, filter-then-traverse approach already does this — resist the urge to widen the window "to be safe." Tune the compaction prompt to preserve derived lineage facts and unresolved mismatches, and clear raw SQL blobs once parsed.

12.1 · To-do list vs progress checker vs notes — the precise distinction

These three are constantly conflated. They answer different questions and point in different temporal directions. Get this right and the whole system clarifies.

DocumentAnswersDirectionStructureUpdated
To-do / checklist"What must happen, and where am I in the sequence?"Forward-lookingItemized: discrete tasks, each with a status (pending → in progress → done)On every item transition

The second axis: in-session vs cross-session

A to-do list comes in two forms. In-session (ephemeral): Claude Code's TodoWrite — a checklist that lives in the context window, gone on reset/compaction; its job is anti-forgetting within a long task (every update returns a reminder nudging the next item). Cross-session (persistent): a file on disk — Anthropic's harness uses a feature_list.json / tests.json checklist and a claude-progress.txt log; Claude Code's newer Task tools (TaskCreate/Update/Get/List) persist to disk with dependencies and hierarchy. The progress log is always cross-session by design — it exists precisely to survive the reset.

12.3 · How files get written and updated — it's a tool call, not a separate function

This is the crux of your question. The model does not call a hidden "diff engine." Updating a memory file is itself a tool call that the model emits, chosen because the system prompt and the tool description told it to. Your client-side handler then executes that call against the real file. Anthropic's Memory Tool (type: "memory_20250818") defines six commands — and the edit primitive is str_replace:

CommandWhat it doesRole
viewList a directory, or read a file with line numbers (optional range)Read / retrieve
createCreate a new file (errors if it already exists)Write (new)
str_replaceFind an exact, unique snippet and swap it for new textEdit — the surgical "diff"
insertInsert text after a given line numberEdit (add)
deleteRemove a file or directoryForget
renameMove or rename a file/directoryReorganise

So "how does the LLM apply something like a git diff and delete a few lines and add a few lines?" — exactly via str_replace: the model emits the old text and the new text, and your handler performs the in-place replacement. The model is deciding what to change (prompt-driven); your code only executes. Two consequences to design around:

  • Verbatim-unique constraint. str_replace only works if old_str appears exactly once. If it's ambiguous, the model must include more surrounding context. (This is the same constraint as a code editor's find-and-replace — and it's why models sometimes re-read a file before editing.)
  • Edit-in-place over rewrite. Prompt the model to prefer str_replace over rewriting whole files; full rewrites waste tokens and risk dropping content. For the structured checklist, restrict edits to a single field.
Don't confuse two different "diffs"

str_replace edits the memory file. Separately, Anthropic's harness has each session commit to git — that's a workspace snapshot enabling rollback/recovery of code, not a memory edit. Use both: str_replace for surgical memory updates; git commits for "revert the last hour of code." They're complementary recovery mechanisms.

12.5 · Procedural & episodic memory — the tool docs you asked about

Unlike semantic facts, these two are rarely a single "write this fact" tool. Their canonical implementations:

  • Procedural memory → a prompt optimiser, not a CRUD tool. LangMem's create_prompt_optimizer takes trajectories (a conversation plus a feedback score) and returns an improved system prompt — the agent literally rewrites its own instructions from experience. Three algorithms: metaprompt (reflect, then propose), gradient (separate critique and proposal steps), and prompt_memory (single-step). In the file world, the equivalent is your skills/*.md / .clinerules being edited (by you or by auto-memory). Docs: LangMem conceptual guide.
  • Episodic memory → a search tool over distilled examples. LangMem stores episodes as few-shot examples ("how a past problem was solved," not just "what the answer was") in a collection, and exposes create_manage_memory_tool (write) + create_search_memory_tool (retrieve by similarity). The "tool" for episodic memory is therefore a retrieval tool that injects the most similar past episodes into the prompt — exactly the pattern in clicode prompt (6).
  • Namespacing for isolation. Every LangMem memory carries a namespace (e.g. ("project","{user_id}","collection")) so memories never leak across users/projects — the file analogue is per-project directories and Claude Code's machine-local, project-scoped MEMORY.md.

12.6 · Architecture — native tool vs MCP server

Two ways to wire all of the above. The decision is about sharing and retrieval richness.

A
Native client-side memory tool
Add the Memory Tool (memory_20250818) to the request; implement handlers for view/create/str_replace/insert/delete/rename against your filesystem (or DB, S3, encrypted store). Simplest path; the SDKs ship base classes to subclass. Best for a single agent with file memory. Must sandbox all paths under /memory and block traversal (../, URL-encoded variants).
B

Wrap the same operations as MCP tools, plus richer retrieval co-located with the data. Choose this when memory is shared across agents/clients (Claude Code, Cursor, your own loop all hitting one memory), or when retrieval needs graph traversal / hybrid search beside the files. Bloat. Without an explicit "keep it coherent, edit don't append, delete the stale" instruction, files grow into noise. Cap file sizes; paginate reads. str_replace ambiguity. 13 · The next level — firewall-proof

The embedded memory engine: one SQLite file, no cloud, no approvals

Here's the honest "next major upgrade" under your constraint, and the good news is it keeps everything you like about markdown. The shift is not a fancier file format or a SaaS — it's a change in the verb: stop making the model read memory; start letting it query memory. The substrate that makes that firewall-proof is one you already have — SQLite — plus, at most, a single vendorable pip package. And you keep your markdown files as the human-readable, git-friendly, auditable source of truth; the database is just a query index layered on top.

What "firewall-proof" actually demands

No runtime network egress · no servers to stand up · standard-library or single vendorable packages (nothing that phones home or needs vendor approval) · single-file portability you can copy or commit anywhere · fully inspectable. Markdown clears that bar. The upgrade has to clear it too — and SQLite does, because sqlite3 ships inside Python's standard library.

13.1 · Why markdown caps out

Files-as-memory has a hard ceiling, and you hit it the moment memory grows past what fits in context:

  • Read-the-whole-file doesn't scale. Retrieval is "open the file" — fine for a 60-line conventions.md, useless once you have thousands of episodes.
  • Grep-only. You can match literal strings but not meaning, and you get no ranking — no "most relevant first."
  • Flat. No relations, no graph, no metadata queries ("show me every failed parse for table X last week").
  • Manual consolidation. The model has to keep files tidy by hand; they bloat.

The upgrade is exactly the move from the agent's filesystem to the agent's database — selective, ranked, structured retrieval instead of dumping files into context (which §02's context-rot argument says is the wrong thing to do anyway).

13.2 · Why SQLite is almost unfairly well-suited

SQLite gives you querying while keeping every property of markdown memory. The field has already converged here — there are several open-source "one SQLite file = agent memory" projects (§14) built on precisely this.

One caveat to verify, not assume

FTS5 ships in the SQLite amalgamation and is present in most macOS/Linux Python builds, but some minimal builds omit it. Verify in one line before relying on it: python -c "import sqlite3; sqlite3.connect(':memory:').execute('CREATE VIRTUAL TABLE t USING fts5(x)')" — if it doesn't raise, you're good. If it's missing, FTS5 is also available as a small loadable extension, or fall back to numpy cosine + LIKE.

13.3 · The three retrieval upgrades — all local

1 · lexical
FTS5 / BM25 — ranked keyword search. Replaces grep. Zero deps.
2 · semantic
Vector — sqlite-vec or numpy cosine. Needs a local embedder.
3 · hybrid
RRF + scoring — fuse both, add recency/importance. Rank, don't dump.
Career bridge · BM25 is the right default for BMG

Every project that benchmarks this finds the same thing: for exact-identifier, technical-term queries — column names, proc names, TF_/EX_ prefixes — BM25 beats dense vector search. Lineage memory is overwhelmingly identifier-driven, so your Tier-0 (FTS5-only) engine is already excellent before you add any embeddings. Add vectors for the fuzzy "what did I do about timeouts" queries; keep BM25 as the backbone.

13.4 · The lineage graph — in the same file, no graph DB

Store edges as a table and traverse with a recursive CTE — your column-tracing, in pure SQL, with no Neo4j and no server. And add Graphiti's bi-temporal trick as a single column: a valid_until that you set (not delete) when a source SQL/Ab Initio file changes, so stale lineage is invalidated while the history stays queryable — the audit-trail property WF governance wants.

-- trace a column back to its source leaves (filter-then-traverse, in SQL)
WITH RECURSIVE upstream(node, depth) AS (
  SELECT :column, 0
  UNION
  SELECT e.src, u.depth+1
  FROM edges e JOIN upstream u ON e.dst = u.node
  WHERE e.valid_until IS NULL AND u.depth < 50)   -- only current edges
SELECT DISTINCT node FROM upstream;

You can also keep NetworkX (pure-Python, pip) in memory for richer algorithms and persist it to the same SQLite file — but for "walk upstream to the leaves," the CTE needs nothing beyond the standard library.

13.5 · The embedding-behind-a-firewall problem — the real wrinkle

Vector search needs an embedding model, and calling a cloud embedding API is exactly what your firewall blocks. Three tiers, easiest first:

0
No embeddings at all
FTS5/BM25 + metadata filters, then let the LLM itself rerank the top-N candidates. Pure standard library — nothing to vendor. For identifier-heavy lineage memory this is often all you need. Start here, and only add vectors when you observe recall failures on fuzzy queries.
1
Vendored local embeddings (the key technique)
A tiny CPU model whose weights you download once on a connected machine, then commit/copy into the repo — so there is zero runtime egress. 13.6 · A concrete schema for clicode — step by step

First, kill the wrong mental model

It is NOT one table per markdown file. It is one table with many rows. Each row is one small chunk of memory (a fact, a bullet, an episode, a convention). The "memory type" is a column on each row — type — not a separate table and not a separate file. The file boundary from §12 (conventions.md vs episodic_log.jsonl) becomes a value in the type column (plus a source column that remembers which file it came from). That is the whole trick: "which file" stops being a location and becomes a filterable field.

Step 1 — Create the store (run once)

One file, clicode_memory.db, sitting in the project next to /memory/*.md. The type values are a small controlled vocabulary you choose; source ties each row back to its markdown file.

PRAGMA journal_mode=WAL;            -- many readers + one writer, safely

CREATE TABLE memory (
  id INTEGER PRIMARY KEY,
  type        TEXT NOT NULL,   -- the type lives HERE: semantic|episodic|procedural|progress|convention
  source      TEXT,            -- which markdown file this row came from (provenance / rebuild)
  content     TEXT NOT NULL,   -- one chunk of text (a bullet, a fact, one JSONL episode)
  table_name  TEXT, column_name TEXT,   -- BMG locators (NULL if not applicable)
  importance  REAL DEFAULT 0,
  ts          TEXT DEFAULT (datetime('now')),
  valid_until TEXT);           -- NULL = current; set (not delete) on invalidation

CREATE INDEX idx_type  ON memory(type);       -- so type filters are instant
CREATE INDEX idx_valid ON memory(valid_until);

-- ranked keyword search over the SAME rows (built in, zero deps)
CREATE VIRTUAL TABLE memory_fts USING fts5(
  content, table_name, column_name, content='memory', content_rowid='id');
-- + INSERT/UPDATE/DELETE triggers to keep memory_fts in sync (add all three as a set)

-- optional (Tier 1+): one embedding per row, stored once
ALTER TABLE memory ADD COLUMN embedding BLOB;   -- numpy float32 bytes; or use sqlite-vec's vec0

CREATE TABLE edges (src TEXT, dst TEXT, rel TEXT, valid_until TEXT);  -- lineage, same file

Step 2 — Ingest the markdown into rows (run once, then on change)

This is the step that was implicit before. A small routine reads each markdown file, splits it into chunks, and inserts one row per chunk — stamping the type from the file's category and embedding the chunk once, here (never again at query time). Re-run it when a file changes (skip unchanged files by content hash).

# map each source file to a memory type
TYPE_OF = {"conventions.md":"convention", "progress.md":"progress",
           "episodic_log.jsonl":"episodic", "skills/":"procedural"}

def ingest(path):
    typ = TYPE_OF[path.name]
    for chunk in split(read(path)):          # a bullet / paragraph / one JSONL line
        emb = embed(chunk)                   # Tier 1+: compute ONCE, store it
        db.execute("INSERT INTO memory(type,source,content,embedding) VALUES(?,?,?,?)",
                   (typ, path.name, chunk, emb))   # the type is written per row

Step 3 — Search (where the type column finally earns its keep)

This is the part you said you couldn't see: the type column used as a filter. The pattern is filter first, score second — narrow to the relevant rows by structured metadata, then rank only that subset.

def search(query, type=None, table=None, k=8, mode="hybrid"):
    # (a) FILTER — the type / locator / validity filter (this is the new power)
    where, params = "valid_until IS NULL", []
    if type:  where += " AND type = ?";       params += [type]
    if table: where += " AND table_name = ?"; params += [table]

    # (b) KEYWORD — FTS5, ranked by BM25, restricted to the filtered subset
    kw = db.execute(f"""
        SELECT m.id, m.content, bm25(memory_fts) AS s
        FROM memory_fts JOIN memory m ON m.id = memory_fts.rowid
        WHERE memory_fts MATCH ? AND {where}
        ORDER BY s LIMIT 50""", [query, *params]).fetchall()
    if mode == "bm25": return kw[:k]

    # (c) SEMANTIC — embed only the QUERY, cosine vs the stored vectors of the SAME subset
    rows = db.execute(f"SELECT id, content, embedding FROM memory WHERE {where}", params).fetchall()
    qv = embed(query)
    vec = sorted(rows, key=lambda r: cosine(qv, unblob(r.embedding)), reverse=True)[:50]

    # (d) FUSE (RRF) + recency/importance, (e) return the top-k CHUNKS (not files)
    return rrf(kw, vec, k=k)

Steps 4–6 — Inject, then maintain

Inject only the top-k returned rows into context (not whole files — that's the §02 context-rot win). Then maintain: graph.trace walks edges via the recursive CTE in §13.4; on a source change you set valid_until on the affected rows (auditable forget); a background pass dedupes and promotes repeated fixes. Crucially, the markdown files stay the source of truth — the table is rebuilt from them by re-running Step 2.

"But isn't this just grep + cosine again? How is it easier?"

Fair challenge — and the honest answer is: the cosine math is identical. What changes is everything around it. Four concrete differences, none of which the bare markdown files give you:

  • Embed once, not every query. With raw files, the vectors live nowhere — you'd re-embed every chunk of every file on each query (slow, model loaded each time), or hand-build a cache and reinvent the database badly. Here the embedding is computed at write time and stored on the row; at query time you embed only the short query string.
  • Keyword search is an indexed, ranked inverted index — grep is an unranked linear scan. Grep reads every byte and returns substring hits in file order. FTS5 jumps to matching rows and ranks them by BM25 relevance. Ranking is the thing grep fundamentally cannot do.
  • Filter-then-score. This is what the type column buys you. "Only episodic memories for table EUCT_X that are still valid" is one WHERE clause that narrows to a handful of rows before you run any similarity. You score a pre-filtered slice, not the whole corpus. With files you'd open, parse, and filter by hand.
  • Record granularity. You retrieve the top-k chunks — the precise high-signal slice — not whole files. Fewer, better tokens in the window.

The honest caveat: at small scale none of this matters — if your whole memory fits in context, just read the files; the database is overkill. The engine earns its place once memory grows past what you can dump in the window, once you need ranking, and once you need to slice by type / locator / validity. That threshold is exactly where a deep-work agent like clicode lives.

13.7 · The tool surface — same shape as §12, now engine-backed

Expose it as MCP tools (clicode already runs an MCP server, so fold these in) or as plain tool calls. The agent's mental model barely changes — it still "writes notes" and "searches memory" — but retrieval is now ranked and selective.

13.8 · Write, consolidate, forget — now that you can query

The engine makes §04's management operations cheap and automatic: dedup at write time by similarity threshold (don't store a near-duplicate); decay by folding recency into the ranking rather than deleting; invalidate via valid_until when source code changes (bi-temporal, auditable); consolidate in a background "sleep-time" routine you run at session end — and promote any fix that has resolved the same class of problem 3+ times into a procedural skill. All local, all in one file.

13.9 · Keep markdown as the source of truth (non-destructive)

This is the architectural elegance and the reason it satisfies your constraint completely. Files stay canonical — human-readable, git-diffable, the audit record a regulator can read. SQLite is a derived index, rebuildable from the files at any time via content-hash sync (re-index only what changed). If the DB corrupts, or you move tools, or someone audits you, you lose nothing — the markdown is intact and the index is regenerated with one command. You are adding a query layer, not replacing your memory.

13.11 · Reference builds & the OSS-framework question

You don't have to write it from scratch — several Apache/AGPL projects are exactly this and run with no cloud: sqliteai/sqlite-memory (markdown-as-source-of-truth + hybrid FTS5/vector + local embedding via llama.cpp), and bozbuilds/AIngram (one SQLite file: sqlite-vec + FTS5 + a knowledge graph + an MCP server, RRF fusion, explicitly "no cloud dependencies"). Read them as blueprints.

"Can't I just self-host Mem0 / LangMem / Letta?"

Yes — they're all pip-installable open source and can run fully self-hosted with a local model, so if pip works and you can point them at a local embedder, the firewall doesn't rule them out. The trade-offs: they pull heavier dependency trees, several default to cloud embedding/LLM endpoints you'd have to override, and they're more to audit. The purest firewall-proof, maximally-portable, fully-inspectable path is the roll-your-own SQLite engine (or the one-file projects above). Pick frameworks for speed-to-build; pick the SQLite engine for portability, auditability, and zero approval surface — which, for a regulated-finance setting, usually wins.

Pitfalls specific to the engine

Embedding egress. The one thing that breaks the firewall promise — vendor the model weights, never download at runtime. FTS sync. If you forget the INSERT/UPDATE/DELETE triggers, search returns deleted/stale rows — add all three as a set. One writer. SQLite allows many readers but one writer; use WAL and keep transactions short. BM25 sign. FTS5's bm25() returns lower-is-better and negates internally — order correctly or your "best" results come last. Don't lose the audit trail. The files are the truth; the DB is disposable — never let the index become the only copy.

14 · Invocation

How the agent calls memory — tools vs MCP, and why SQLite beats markdown at scale

Two questions remain: how does the agent actually reach the memory (plain tool calls or an MCP server?), and why is the SQLite engine really better than markdown when, as you rightly note, both can filter by date and both can return results? Here is the single reference, for both designs.

14.1 · Tool calls and MCP are not opposites

This trips everyone up. An MCP server is a way to deliver tools; it is not an alternative to tool calls. From the model's point of view both are identical: it sees a tool with a name, a description, and a JSON argument schema, and it emits a tool_use. The argument schema is the same whether you host it natively or behind MCP — MCP does not change the arguments, it changes who runs the handler and who can reach it.

The real axis is where the handler executes:

  • Native (in-process) tools. You put the tool definitions directly in your API request's tools array; when the model calls one, your own application code runs the handler. Fewest moving parts; one app owns the memory.
  • MCP server. The tools live in a separate process speaking the Model Context Protocol; any MCP-capable client (Claude Code, Cursor, your Python loop) can connect and call them. Reusable and decoupled, at the cost of running a server.

14.2 · The markdown design — tools & arguments

For markdown, the idiomatic surface is one tool with a command discriminator — exactly what Anthropic ships as the native memory_20250818 tool, so you often don't even author the schema; you enable the built-in tool and implement six handlers. The arguments per command:

14.5 · Markdown vs SQLite, head to head — answering your challenge directly

You're right on the narrow point: both can filter by date, and both can return results. So the question isn't whether you can — it's where the work happens. That single reframe is the whole answer.

The reframe: selection in the window vs selection in the engine

With markdown, every non-trivial retrieval requires loading text into the context window first, and then the model does the filtering/ranking by reading it. With SQLite, the filtering/ranking happens in the engine, before anything touches the window. Since §02 established that the context window is the scarce, rot-prone resource, moving selection out of it is the entire advantage. "You can filter by date in both" hides this: in markdown you filter by date by spending context on the whole file and trusting the model to eyeball dates; in SQLite a WHERE ts > ? returns only the matching rows at zero context cost.

The concrete case that makes it undeniable. clicode has 4,000 episodic entries across many sessions — hundreds of KB, far past any sane context budget — and is about to trace EUCT_TAX.subvention. It wants: a successful subvention-mapping fix for this table in the last month.

  • Markdown: it can't load 4,000 lines. It greps subvention → ~60 unranked lines mixing successes and failures and old and new; it loads those 60 lines (context cost), eyeballs dates and outcomes in-prompt, and misses the relevant entry that was logged as "premium allocation" (no shared substring).
  • SQLite: one episodic_search with since + outcome='success' + table filters to a handful of rows, ranks them by BM25+cosine, and returns the top 3 — including the "premium allocation" one (semantic match) — for ~300 tokens.

Same "filter by date," available in both — but in markdown it costs the whole file in context, relies on in-prompt date reasoning, can't rank, and can't match meaning; in SQLite it's exact, ranked, semantic, and nearly free of context.

The honest verdict

It is not that SQLite computes better similarity math — cosine is cosine. It is that SQLite retrieves without spending context, with ranking, and with aggregation — three things "read the file" and grep fundamentally lack. And the tell: the moment you bolt a real search index onto markdown to "retrieve equally well," you have built the SQLite approach — pure markdown's only retrieval primitives are read-file and grep, and both break past the window. So: small memory → markdown wins (simpler, auditable, zero deps); memory past the window → SQLite wins, because in markdown retrieval is "load candidates into context," which is the exact resource you're trying to conserve.

14.6 · The advantages of the engine — best reasons first

If you keep one ranked list from this whole document, keep this one. Why a local SQLite engine beats flat markdown memory, most decisive reason first:

  1. It answers questions the window can't hold. Aggregations, ranked top-k, and dependency traversals over a corpus larger than the context window are impossible with read-file + grep — grep can't compute, and the model can't see it all at once. SQLite returns the answer (a count, a top-3, a traced path) without putting the corpus in context. This is "cannot," not "slower" — the single decisive reason.
  2. Selection happens in the engine, not in the window. Filtering and ranking run before anything enters the context window, conserving the scarce, rot-prone resource. Markdown can only filter by first spending context on the candidates.
  3. It ranks. It returns the top-k most relevant records, so you inject ~3 high-signal rows instead of 60 unranked grep lines or whole files. Grep has no notion of relevance at all.
  4. It matches meaning, cheaply. Embeddings stored once let "hang" find "timeout." Markdown either can't do semantic search, or must re-embed the entire corpus on every query — which is just rebuilding the database, badly.
  5. It filters by structured metadata. type, table/column, validity, date — pushed down so you score only the relevant slice. Exact, and free of context cost.
  6. It aggregates and joins. COUNT / GROUP BY / JOIN answer "how many columns are unverified?" or "which fix recurs across sessions?" — questions markdown can only approach by loading everything and reasoning in-prompt.
  7. It scales and stays clean. Sublinear index lookups, dedup at write time, bounded growth, and bi-temporal valid_until for auditable forgetting. Flat files bloat and slow linearly, and have no cheap way to detect contradictions.

The honest counterweight: below the window, none of this matters — if all your memory fits in context, markdown wins on simplicity, auditability, and zero dependencies. The engine earns its keep precisely when memory outgrows the window, which is exactly where a deep-work agent like clicode lives.
